# Explaino-all Launch Anchor

Use this file as the first checked-in authority surface for any future `Explaino-all` implementation prompt.

Purpose:

- keep future prompts short without making them underspecified
- force kickoff through checked-in repo authority instead of chat reconstruction
- preserve the Explaino-family canonicalization invariants even if the exact slice kickoff wording changes later

## Non-Negotiable Identity Rules

- `Explaino-all` is one canonical public Explaino family identity, not another peer variant beside the old list.
- Old Explaino variants are legacy selectors, presets, or compatibility aliases; they are not the long-term architectural center.
- The public identity stays neutral. Named theme/variant shorthand may exist internally, but it must project through canonical `Explaino-all` axis state.
- Zero or neutral axis values must collapse back to the correct baseline Explaino behavior truthfully.
- Adding a new Explaino family axis must update one canonical registry and flow everywhere from there; it must not require copy-pasted selector/default/UI/runtime wiring across disconnected files.

## Scope Fence

Future `Explaino-all` slices may:

- canonicalize Explaino family identity
- unify Explaino family axis exposure
- project legacy variant selectors into canonical axis state
- add enforcement guardrails that stop future Explaino drift

Future `Explaino-all` slices must not:

- reopen shipped generic advanced-color core work
- reopen historical `234919_563__explaino_inertial` compatibility work unless explicitly reclassified
- widen into unrelated new fractal families
- treat `Explaino-all` as a reason to skip proof, stale-plan gates, or slice contracts

## Slice Order

Follow this order unless live repo evidence proves a narrower truthful split is better:

1. canonical identity plus axis registry
2. legacy variant projection plus preset/alias truth
3. enforcement guardrails
4. optional cleanup only if the first three slices leave truthful residual migration debt

Do not start with enforcement before the canonical identity exists.
Do not start with cleanup before legacy projection truth exists.

## Per-Slice Success Rules

### Slice 1 - Canonical Identity + Axis Registry

Must prove:

- `explaino_all` exists
- it is first in the Explaino list
- it is the default selection on load
- all current variant axes are represented in one shared canonical registry
- the `Explaino-all` control surface is data-driven from that registry

### Slice 2 - Legacy Variant Projection + Preset Truth

Must prove:

- legacy Explaino selectors map to canonical `explaino_all` axis state truthfully
- runtime, persistence, reset, and diagnostics surfaces agree
- old labels do not preserve a second hidden implementation authority

### Slice 3 - Enforcement Guardrails

Must prove:

- a new Explaino axis cannot land without registry coverage
- a new Explaino alias/preset cannot land without canonical projection coverage
- canonical neutral/default invariants remain enforced

## Post-Closeout Gate

Every future `Explaino-all` slice must re-read its active phased plan on the committed head and fail closed if the plan still contains stale pre-closeout language or untruthful open asks.
